# Runbook: Merger / Core System Conversion

> **Mode**: CANVAS-Full | **Duration**: 12-24 months | **Agents**: All | **Lead**: CEO + CFO

---

## Scenario

The credit union is undertaking a merger with another credit union (as either acquirer or acquired institution), or is converting its core processing system to a new platform. Both scenarios require coordinated execution across every functional area, with zero tolerance for member disruption, compliance gaps, or data integrity failures. This runbook covers both scenarios with a shared framework and scenario-specific branches.

## When to Activate

**Merger Scenario**:
- Board has approved a letter of intent to merge with another credit union
- The credit union is the surviving institution (acquirer) or the merging institution
- NCUA merger application is being prepared or has been filed

**Core Conversion Scenario**:
- Board has approved a new core processing system contract
- Implementation planning is beginning with the new core vendor
- Parallel processing period is being planned

## SCENARIO A: Merger Execution

### Phase A-1: Due Diligence and NCUA Application (Months 1-4)

```
MONTH 1: Due Diligence Launch
├── CEO: Engage due diligence team — assign functional leads to each area
├── CFO: Financial due diligence
│   ├── Review merging CU audited financials — 3 years
│   ├── ALLL adequacy assessment — loan loss reserve validation
│   ├── Capital ratio projections — combined institution net worth
│   ├── Concentration risk analysis — combined loan and deposit portfolios
│   └── Cost savings and synergy modeling
├── Compliance Officer: Regulatory due diligence
│   ├── Prior examination history — MRAs and findings
│   ├── Active regulatory orders or informal agreements
│   ├── Pending member complaints or litigation
│   └── Compliance management system assessment
├── BSA Officer: BSA/AML due diligence
│   ├── Prior BSA examination results
│   ├── SAR filing history and open investigations
│   └── AML risk profile assessment
└── Risk Manager: Portfolio due diligence
    ├── Loan quality review — sample 50+ files
    ├── Delinquency and charge-off trend analysis
    └── Operational risk assessment

MONTH 2: Due Diligence Completion
├── CFO: Consolidated financial model — combined balance sheet and income statement
├── Risk Manager: Final risk assessment report — known risks and mitigation plans
├── Compliance Officer: Compliance risk assessment — remediation required before or after merger
├── Internal Auditor: Independent validation of due diligence findings
└── CEO: Due diligence report to board — proceed / renegotiate / walk away

MONTHS 3-4: NCUA Application and Member Vote
├── CEO + Compliance Officer: Prepare NCUA merger application
│   ├── Combined CU business plan and financial projections
│   ├── Management and governance plan
│   ├── Member benefit disclosure (required)
│   └── Charter, bylaws, and field of membership documentation
├── CEO: Member notice — NCUA-required notification to both CU memberships
├── CEO: Member vote conducted (if required by charter)
├── NCUA: Application review period (typically 60-90 days)
└── CEO: NCUA approval received — proceed to integration planning
```

### Phase A-2: Integration Planning (Months 4-8)

```
MONTH 4-5: Integration Blueprint
├── Core Systems Administrator: Core system integration plan
│   ├── Which core system survives — evaluate both platforms
│   ├── Data migration scope — member, loan, deposit, card data
│   ├── Integration timeline with conversion date
│   └── Parallel processing plan
├── Deposit Operations Manager: Deposit portfolio integration plan
│   ├── Account type mapping — merging CU products to surviving CU products
│   ├── Member notification plan — account number changes if applicable
│   └── Rate harmonization plan (ALCO input)
├── Loan Officer: Loan portfolio integration plan
│   ├── Loan type mapping and policy differences
│   ├── Underwriting policy harmonization
│   └── Servicing continuity plan
└── CHRO: Staff integration plan
    ├── Organizational structure of combined institution
    ├── Retention offers for key merging CU staff
    ├── Redundancy planning (if applicable)
    └── Culture integration approach

MONTH 6-8: Member Communication and Retention
├── Marketing Manager: Member communication campaign
│   ├── Why the merger serves members
│   ├── What changes (products, rates, offices, contact info)
│   ├── What stays the same (accounts, loans, service quality)
│   └── How to get answers — dedicated FAQ page, branch events
├── Financial Advisor: Proactive outreach to merging CU high-value members
├── Branch Manager: Branch integration plan — hours, staffing, signage
└── Member Services Representative: Prepared for member questions and concerns
```

### Phase A-3: Legal Close and Day 1 Integration (Month 8-12)

```
MONTH 8-10: Systems Integration Execution
├── Core Systems Administrator: Execute data migration (test environment first)
├── Core Systems Administrator: Parallel processing period — both systems live
├── Core Systems Administrator: Data validation — every account, every loan, every balance
├── Internal Auditor: Independent data validation sign-off
└── All agents: Testing in new combined environment before cutover

MONTH 11-12: Legal Close and Post-Merger
├── CEO + CFO: Legal merger close — effective date
├── Core Systems Administrator: Core cutover on conversion date
├── All agents: Day 1 operations on combined platform
├── Member Services Representative: Surge staffing for member inquiries Day 1-7
├── CFO: First combined financial reporting cycle
└── Compliance Officer: Combined institution compliance program activation
```

## SCENARIO B: Core System Conversion

### Phase B-1: Vendor Selection and Contract (Months 1-3)

```
MONTH 1-2: Vendor Evaluation
├── Core Systems Administrator: Lead RFP process — functional requirements defined
├── CFO: Total cost of ownership analysis for each vendor
├── Compliance Officer: Regulatory compliance features evaluation
│   ├── TRID, TILA, Truth in Savings disclosure generation
│   ├── BSA/AML monitoring and CTR/SAR reporting
│   ├── HMDA data capture and reporting
│   └── Reg E and Reg CC processing
├── Risk Manager: Vendor risk assessment — SOC 2, financial stability, reference checks
└── CEO: Vendor selection recommendation to board — board approval required

MONTH 3: Contract Negotiation
├── CEO + CFO: Contract negotiation — implementation timeline, fees, SLAs, data ownership
├── Core Systems Administrator: Technical specifications and data migration scope
├── Compliance Officer: Contract review for regulatory compliance obligations
└── CEO: Board approval of final contract
```

### Phase B-2: Implementation Planning (Months 4-8)

```
MONTHS 4-5: Project Structure
├── Core Systems Administrator: Project lead and conversion manager
├── CEO: Executive sponsor; monthly status to board
├── CHRO: Training coordinator for all staff on new system
├── Each functional agent: Designated system champion for their area
└── CFO: Conversion budget tracking and vendor milestone payments

MONTHS 6-8: Data Mapping and Configuration
├── Core Systems Administrator: Data mapping — old system fields to new system fields
│   ├── Member/account master data
│   ├── Loan data — all products, all fields
│   ├── Deposit data — all products, all fields
│   ├── Transaction history — retention scope
│   └── Card data — coordinate with card processor
├── Deposit Operations Manager: Deposit product reconfiguration in new system
├── Loan Officer: Loan product reconfiguration in new system
├── Compliance Officer: Disclosure document configuration and accuracy testing
├── BSA Officer: AML monitoring and CTR/SAR workflow configuration
└── Core Systems Administrator: Interfaces and integrations — card processor, ACH, Fed, SWIFT
```

### Phase B-3: Testing and Training (Months 9-14)

```
MONTHS 9-11: Parallel Testing
├── Core Systems Administrator: Test environment configured with production data copy
├── Each functional agent: Test their workflows in new system
│   ├── Member Services Rep: Account opening, maintenance, Workflow A
│   ├── Loan Officer: Loan origination, Workflow B
│   ├── Mortgage Processor: TRID processing, Workflow C
│   ├── Collections Specialist: Collections workflows, Workflow D
│   ├── Deposit Operations Manager: Deposit operations, Workflow E
│   └── BSA Officer: CTR/SAR workflows
├── Compliance Officer: Disclosure output testing — accuracy against regulatory requirements
└── Internal Auditor: Independent testing review — critical functions validated

MONTHS 12-14: Staff Training
├── CHRO: Training curriculum developed — role-based modules
├── Core Systems Administrator: Train-the-trainer program with vendor
├── Each functional lead: Train their teams
├── Branch Manager: Branch staff training — teller, MSR, loan officer workflows
└── CHRO: 100% training completion required before go-live date
```

### Phase B-4: Conversion Weekend and Post-Live (Months 15-18)

```
CONVERSION WEEKEND
├── Core Systems Administrator: Final data extract from old system — end-of-day Friday
├── Core Systems Administrator: Data migration to new system — automated migration + manual validation
├── Core Systems Administrator: Parallel run — both systems balanced
├── CFO + Deposit Operations Manager: Balance verification — every account, every GL
├── Core Systems Administrator: Go/No-Go decision — Sunday 6pm before Monday live
└── Branch Manager: Staff briefing Sunday — new system live Monday morning

FIRST WEEK POST-CONVERSION
├── Core Systems Administrator: All-hands support — peak issue resolution period
├── Member Services Representative: Surge staffing — member inquiry volume will spike
├── Branch Manager: Branch managers on floor, not in offices — visible leadership
├── Core Systems Administrator: Issue log — document every problem and resolution
├── CFO: Daily GL reconciliation — catch any migration errors immediately
└── Compliance Officer: Monitor for any disclosure or processing errors

MONTH 16-18: Stabilization
├── Core Systems Administrator: Issues resolved and documented; old system decommissioned
├── Compliance Officer: Post-conversion compliance review — disclosures accurate in production
├── Internal Auditor: Post-conversion audit — data integrity validation
├── CFO: Conversion budget reconciliation — final cost vs. plan
└── CEO: Post-conversion report to board
```

## Shared Quality Gates (Both Scenarios)

### Data Integrity Gate (Critical — No Exceptions)
**Gate Keeper**: Core Systems Administrator + CFO + Internal Auditor

| Criterion | Threshold | Evidence Required |
|-----------|-----------|-------------------|
| Every member account balance matches source | 100% — zero variance | Balance reconciliation report |
| Every loan balance matches source | 100% — zero variance | Loan reconciliation report |
| Every transaction history record transferred | 100% for required retention period | Migration report with record counts |
| All GL accounts balanced | 100% — penny-perfect | Trial balance comparison |
| All interfaces operational | Card, ACH, Fed, digital banking | Interface test results |

**Decision**: If any criterion fails — STOP CONVERSION. Fix and re-test before going live.

### Compliance Continuity Gate
**Gate Keeper**: Compliance Officer + BSA Officer

| Criterion | Threshold | Evidence Required |
|-----------|-----------|-------------------|
| All required disclosures generated accurately | 100% tested against 12 CFR requirements | Compliance test results |
| BSA/AML monitoring active and configured | Verified in new environment | BSA Officer testing sign-off |
| CTR and SAR workflows operational | End-to-end tested | BSA Officer test results |
| HMDA data capture functional | Mortgage loan test confirms all fields captured | HMDA test report |
| Adverse action notices generate correctly | Template tested for all decision types | Compliance Officer sign-off |

### Member Experience Gate
**Gate Keeper**: Branch Manager + Member Services Representative

| Criterion | Threshold | Evidence Required |
|-----------|-----------|-------------------|
| All member-facing staff trained and confident | 100% training completion | CHRO training records |
| Member account access functional | Online banking, mobile app, debit cards tested | Core Systems Administrator test results |
| Branch workflows operational | All common transactions tested | Branch Manager sign-off |
| Member communication delivered | Account number changes, access changes | Marketing Manager delivery log |

## Key Decision Points

| Decision | Timing | Authority | Criteria |
|----------|--------|-----------|---------|
| Proceed with merger vs. walk away | Month 2 (due diligence complete) | Board | Due diligence findings support financial and compliance rationale |
| Go/No-Go on conversion weekend | Sunday 6pm before go-live | CEO + Core Systems Administrator | All quality gates passed |
| Extend parallel processing period | If issues found in testing | CEO + Core Systems Administrator | Data integrity or compliance concerns |
| Rollback conversion | If critical failure post-go-live | CEO + Core Systems Administrator | Member impact unacceptable; rollback plan pre-staged |
| Delay merger close | If data migration fails | CEO + CFO | Member protection requires additional preparation |

---

## Common Pitfalls

| Pitfall | Prevention |
|---------|-----------|
| Data migration errors not caught until after go-live | Balance reconciliation is mandatory — not optional — before go-live |
| Staff trained on system but not on procedures | Train on new workflows, not just new screens |
| Compliance disclosures inaccurate in new system | Compliance Officer tests every disclosure template before production |
| Members lose access to accounts on conversion weekend | Core Systems Administrator tests all access channels; surge support staffing |
| BSA monitoring gaps during transition | BSA Officer validates AML monitoring active before go-live |
| HMDA data capture fails silently | Mortgage team runs test applications before live mortgage originations |
